# 现象描述

当前自动诊断页面左侧信息区按以下顺序叠放：

1. 自动诊断模式 / 统计框
2. 传感器采集周期框
3. 各机组诊断状态框
4. 当前诊断框
5. 待处理故障队列
6. 最近诊断结果

该信息架构对首次使用者不够直观，通常需要额外解释才能理解“先看哪里、每一块表达什么”。
尤其是“各机组诊断状态”目前更接近 cooldown / 可诊断语义，而不是用户最关心的“该机组当前是否仍存在未消缺故障”。

# 预期行为

自动诊断左栏应更贴近用户的自然关注顺序，自上而下建议调整为：

1. 自动诊断模式 / 统计框
2. 各机组诊断状态
3. 传感器采集周期
4. 待处理故障队列
5. 当前诊断
6. 最近诊断结果

同时补充以下语义要求：

- “各机组诊断状态”应优先表达**机组当前故障关注状态**，而不只是 cooldown
- 只要机组存在尚未消缺的故障，应显示红色或高风险态
- 机组一直未发生故障时才显示绿色
- 第一轮诊断尚未结束前，应显示中性 / 无色状态，而不是默认绿色
- “当前诊断”建议补充时间戳，帮助用户理解这是正在处理的哪一批故障
- “传感器采集周期”后续应支持与机组状态联动：点击机组卡片后切换查看对应机组的采集进度

# 复现步骤

1. 打开自动诊断页面
2. 观察左侧信息区各模块从上到下的顺序
3. 尝试基于当前页面判断：
   - 哪台机组最值得关注
   - 当前是否仍有未处理故障
   - 当前诊断对应哪一批输入
4. 对比现有展示与用户自然认知路径

# 根因分析

原 `UnitCooldownGrid` 组件以 `cooldown` 剩余秒数为主要语义：有冷却时显绿色（可诊断），无冷却时也显绿色（就绪）。第一轮 epoch 前 `epoch_num=0` 时所有机组已默认绿色，与”系统刚启动尚未采集数据”的实际状态相悖。同时，`pending_queue` 中的待处理故障单未与机组状态视觉联动，用户无法从机组格上直接判断哪台机组存在未消缺问题。

信息卡片的原有排序（统计框 → 传感器采集周期 → 机组状态 → 当前诊断 → 待处理队列）将”传感器采集周期”置于”机组故障状态”之前，导致用户首先看到的是技术细节而非关键运营状态。

# 修复方案

1. **语义重构**：将 `UnitCooldownGrid` 重命名为 `UnitStatusGrid`，驱动逻辑改为：
   - `epoch_num=0`：全部显示中性灰色 + `—`（首轮尚未完成）
   - `pending_queue` 中存在该机组：显示红色 + `⚠ 待处理`
   - 否则：显示绿色；若有冷却剩余则附加 `⏱ Xs` 计时器

2. **信息层级重排**：左栏渲染顺序调整为：
   统计框 → 各机组故障状态 → 传感器采集周期 → 待处理故障队列 → 当前诊断 → 最近诊断结果

3. **时间戳补充**：`CurrentDiagnosisCard` 右上角显示 `started_at` 的相对时间（`relativeTime(current.started_at)`），帮助用户判断当前诊断对应的批次。

# 回归测试

> - 自动诊断面板信息区按新的优先级顺序渲染
> - 各机组状态首轮为中性态、存在未处理故障时为红态、无故障时为绿态

# Agent Notes

你的理解基本正确：

- 第一块应保留为全局汇总信息
- 第二块应该是“用户先看哪台机组最危险/最需要处理”，所以要比采集周期更靠前
- 第三块更像第二块的下钻详情，未来做成“按机组切换采集进度”是合理演进
- 第四到第六块应按“待处理历史 → 当前分析 → 已归档结果”或“待处理 → 当前 → 归档”的用户任务流组织，而不是纯技术实现顺序

需要补充的一点是：当前名为“各机组诊断状态”的卡片，实际上如果仍继续用 cooldown 数据驱动，就算换到更上面也会继续误导用户。因此这条 Bug 不只是换顺序，也包含状态语义修正。
