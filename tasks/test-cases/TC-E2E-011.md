## 测试目标

验证当第一条诊断结果处于 pending 未处理状态时，用户发起第二条诊断，系统不会无声地丢失其中任何一条。

## 前置条件

- `POST **/diagnosis/run` → 两次调用分别返回 `#1机` 和 `#2机` 的完整 SSE
- 第一条诊断已点击"稍后处理"（或仍在当前会话中展示）
- 标准 API mock

## 操作步骤

（取决于 PM 选择的行为，见下方产品决策说明）

1. 提交第一条诊断（`#1机`），等待完成
2. 不点击"稍后处理"，直接发起第二条诊断（`#2机`）
3. 等待第二条完成
4. 断言两条结果均可访问（具体断言取决于选定方案）

## 产品决策说明（⚠️ 待 PM 确认，选其一后进 Playwright 实现）

**方案 A**：第二条诊断发起时，第一条自动转入待处理列表（类似 AutoDiagnosis 路径）
- 断言：`/history` 待处理 tab 中可见 `#1机`，当前页面展示 `#2机` 结果

**方案 B**：保持可切换列表，两条结果并列展示
- 断言：UI 中同时可见 `#1机` 和 `#2机` 两个结果卡片或切换控件

**方案 C**：弹出提示，要求先处理第一条才能发起第二条
- 断言：第二次点击"开始诊断"后出现阻断性提示，`#2机` 诊断未触发

## 预期结果 / 不应发生

- 无论哪种方案，`#1机` 结果不能无声消失
- 不应发生：两条诊断都不可见、第一条被静默覆盖
